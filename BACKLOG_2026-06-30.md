### B4. Recipe category organisation — FULL PROPOSAL (2026-06-30, written with user, not yet implemented)

**Root cause, confirmed by reading the actual import code (`deployment/flask_app.py` line ~2250-2266):**

There are currently **two unrelated category systems** that don't talk to each other:

1. **The household's real category list** (`data/categories.json`): meaningful dish-type buckets — Vegetarian, Fish & Seafood, Meat, Other, Homemade, Quick Dinners, Pasta & Noodles, Chicken, Ground Meat & Sausages, Taco & Tex-Mex, Grill, Soups & Stews. These are the categories shown in the navbar dropdown and used for menu generation filtering.
2. **"Imported" pseudo-categories**: `categories.json` also has 5 entries flagged `"imported": true` — Popular Norwegian Recipes, European Classics, Nordic Classics, Holiday Recipes, Summer Recipes. These aren't dish types, they're **recipe-pack names wearing a category costume**.

**The actual bug:** when a recipe pack is imported (`api_recipe_packs_import`), the code explicitly does this to every recipe in the pack:
```python
# Override category with pack name
r['category'] = pack_category_name
```
This **destroys** whatever the recipe's original `category` field was (Meat/Fish/Other/etc.) and replaces it with the pack's name. So a "Greek Salad" imported from the Summer Recipes pack becomes `category: "Summer Recipes"` — it can never show up under a dish-type category like "Vegetarian" or a future "Salads" category, because that information was thrown away at import time. This is exactly the bug the user found: imported recipes only ever live under their pack name, never under a dish-type bucket.

#### B4b. Fix the import-time category destruction (the actual bug)

**Problem:** importing a pack overwrites every recipe's `category` with the pack name, discarding the original dish-type category.

**Proposed fix:** stop overwriting `category` entirely. Instead:
- Keep the recipe's own dish-type `category` (e.g. "Chicken", "Salads") intact through import — this is what should drive normal category-dropdown filtering and menu generation.
- Track *which pack a recipe came from* as **separate metadata**, not as the category. Add a `source_pack` field (e.g. `"source_pack": "summer_recipes"`) purely for "Manage Recipe Packs" bookkeeping (so a user can still see/remove "everything I imported from Summer Recipes" as today), but this field should NOT be what the category dropdown filters by.
- Result: a Greek Salad imported from Summer Recipes shows up under **Salads** (and/or Vegetarian) in the normal category dropdown, exactly as the user wants, while "Manage Recipe Packs" can still group/remove by `source_pack` separately.

**This directly fixes the literal example given:** "if we import norwegian menu pack and it includes chicken recipes, they should also be listed under the chicken category and not just under norwegian menu pack" — yes, exactly this, once B4b is implemented.

---

#### B4c. Add a proper "dish type" tag system, not just one flat category field

**Problem:** today a recipe can only ever be in exactly one category (one string field). Real dishes often fit more than one useful filter at once — e.g. a chicken stir-fry is both "Chicken" and "Quick Dinners" and arguably "Asian-inspired."

**Proposed model:** keep `category` as the *primary* dish-type bucket (one value, drives the main dropdown), but add a `tags` array (the data already has one, currently used loosely for things like `"traditional"`, `"quick"`, `"comfort-food"` — formalize it) that can ALSO be used for cross-cutting filters like:
- `quick` (under 30 min) → powers "Quick Dinners" as a secondary filter without needing its own category
- `cuisine:italian`, `cuisine:mexican`, `cuisine:thai` etc. → powers a future "Around the World" browsing view (see F5) without needing a separate pack per cuisine
- `kid-friendly`, `meal-prep-friendly`, `one-pot` → future filter ideas, not built yet

This turns "category" + "tags" into a proper two-axis system: one primary bucket (what dropdown you see it under) + flexible secondary labels (what else it's good for), rather than trying to cram everything into one field or multiplying categories endlessly.
