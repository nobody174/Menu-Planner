### B14. Shopping list includes prep instructions instead of just the ingredient — FIXED
- **Found:** Mobile testing, 2026-07-03
- **Cause:** Ingredient names in recipe data carry cutting/prep descriptors (e.g. "Gulrot, i skiver" / "Carrot, sliced", "Agurk (tynne skiver)"), which were passed straight through into the shopping list unparsed. Also meant e.g. "Gulrot" and "Gulrot, i skiver" deduplicated as separate lines instead of merging.
- **Fix (2026-07-03):** Added `strip_prep_descriptors()` in `core/ingredient_deduplicator.py` — regex-strips known Norwegian/English cutting/prep words from parenthetical `(...)` and trailing `, ...` suffixes (skiver, biter, terninger, hakket, sliced, diced, chopped, grated, etc.), applied before the dedup key is built (`deduplicate_ingredients`) and again when building the Norwegian name map in `deployment/flask_app.py:shopping_list()`
- **Files:** `core/ingredient_deduplicator.py`, `deployment/flask_app.py`
- **Status:** needs re-test. Note: only affects newly-generated shopping lists — an already-generated menu's cached shopping list won't retroactively update until the menu is regenerated.

### B15. Shopping list uses English unit "pcs" instead of Norwegian "stk" — FIXED
- **Found:** Mobile testing, 2026-07-03
- **Cause:** `pcs` (the English unit string baked into recipe data) wasn't in the deduplicator's unit-normalization list, so it fell through unnormalized instead of becoming "pieces" like `stk`/`piece` did. Norwegian display also never translated the unit at all (only the ingredient name).
- **Fix (2026-07-03):** Added `pcs`/`pieces` to `UNIT_CONVERSIONS` and the primary-unit normalization check in `core/ingredient_deduplicator.py`; added unit translation (`pieces`/`pcs` → `stk`) alongside the existing name translation in `deployment/flask_app.py:shopping_list()`. Also added a space between quantity and unit in `shopping.html` (was rendering "2stk" with no gap).
- **Files:** `core/ingredient_deduplicator.py`, `deployment/flask_app.py`, `frontend/templates/shopping.html`
- **Status:** needs re-test

### B17. App requires relogin after installing as PWA / being away for a while — partially fixed, still recurring
- **Found:** Mobile testing, 2026-07-03
- **Cause 1 (fixed):** Flask session cookie was browser-session-only (no `PERMANENT_SESSION_LIFETIME`, `session.permanent` never set), so closing an installed PWA (common on mobile) dropped the session
- **Fix 1 (2026-07-03):** Added `PERMANENT_SESSION_LIFETIME = 30 days` and `session.permanent = True` on both local and Azure login paths (`deployment/flask_app.py`)
- **Still happening (2026-07-04):** User reports still getting logged out often on phone, alongside intermittent "Oops!" server error pages that clear on refresh. `FLASK_SECRET_KEY` was confirmed already set as a stable Render env var, so it's not a key-rotation-on-restart issue.
- **Cause 2 (likely, fixed):** The Postgres engine (`database/database.py`) had no `pool_pre_ping`, so the first request after the app/DB has been idle (Render spins services down on inactivity) reused a dead connection and threw a 500 - which is very plausibly both the "Oops!" error *and* an indirect cause of session flakiness (a failed DB lookup mid-request can misbehave depending on what it was doing).
- **Fix 2 (2026-07-04):** Added `pool_pre_ping=True` and `pool_recycle=280` to the Postgres engine in `database/database.py` so stale connections are transparently replaced instead of erroring.
- **Status:** needs re-test over a few days of normal use — if logouts/Oops still happen after this, the PWA-specific angle (iOS WebKit's ITP can purge storage for standalone home-screen apps not opened as a real Safari tab within ~7 days) may be worth investigating next, that's a platform limitation rather than something fixable server-side.

### B19. PIN box auto-focused/auto-opened when entering household/settings on mobile — FIXED
- **Found:** Mobile testing, 2026-07-03
- **Cause:** Both the PIN and password inputs in the owner-auth modal (`profile-picker.html`) carried the raw HTML `autofocus` attribute unconditionally, even though the modal itself is hidden (`display:none`) until the user clicks the Owner tile. Some mobile browsers still fire autofocus (and pop the on-screen keyboard) on a hidden input at page load. JS already existed (`focusOwnerAuthInput()`) to focus the input correctly only once the modal is actually shown — the static attribute was redundant and the buggy part.
- **Fix (2026-07-03):** Removed `autofocus` from both inputs; focus is now only ever set via JS when the modal is genuinely visible.
- **File:** `frontend/templates/profile-picker.html`
- **Status:** needs re-test on device

### B20. Mixed/wrong measurement units in Norwegian recipe data — FIXED, confirmed
- **Found:** User + testing feedback, 2026-07-03
- **Cause:** Recipe pack data (`data/recipe-packs/*.json` etc.) had inconsistent/English unit strings leaking into the Norwegian `unit.no` field — "tbsp"/"tsp" instead of "ss"/"ts", plus "spiseskje", "spiseskjeer", "teskje", "tsk", "spsk" all meaning the same thing but never normalized to one form, "after taste"/"efter smak" instead of "etter smak", and English size words "medium"/"small"/"large"/"bunch" instead of "middels"/"liten"/"stor"/"bunt". Also one Swedish "skivor" instead of Norwegian "skiver".
- **Fix (2026-07-03):** Ran a scripted normalization pass across all 13 recipe data files (12 recipe packs + sides-stash) using a canonical mapping table — 214 unit entries corrected. Only touched the `unit.no` field via exact-match mapping, nothing guessed/translated freeform.
- **Root cause of user still seeing "pcs"/"tbsp" after that fix:** households that had already imported a recipe pack before the fix landed have their own permanent copy of that recipe data in their database row (`household.recipes_db`) — fixing the seed JSON files only changes what *future* imports get. Already-imported recipes stayed stale.
- **Fix 2 (2026-07-04):** Added `normalize_no_unit()` / `UNIT_MAP_NO` as a shared, reusable mapping in `core/ingredient_deduplicator.py`, and a new admin maintenance action ("Normalize Recipe Units for All Households" button on `/admin`, backed by `/admin/normalize-recipe-units`) that re-applies the same normalization directly to every household's already-imported recipe data in the database, with a confirmation prompt.
- **Files:** `core/ingredient_deduplicator.py`, `deployment/flask_app.py`, `frontend/templates/admin.html`
- **Not fixed — needs manual review:** 2 combined-ingredient rows with genuinely ambiguous units that a mechanical mapping can't safely resolve:
  - `pack_pasta_noodles.json` recipe `eu_096` — "Béchamel (melk, smør, mel)" has unit `ml/100g/50g` (three ingredients bundled into one line with one amount)
  - `pack_beef_red_meat.json` recipe `eu_083` — "Smør og melk" has unit `gram/100ml` (same issue)
  - These need to be split into separate ingredient lines with correct individual amounts, not just relabeled.
- **Confirmed fixed:** 2026-07-04 — admin ran "Normalize Recipe Units for All Households", backfilled 370 ingredient units across both households

### B21. "Change day" (swap recipe) on All Recipes page says success but doesn't actually change anything — FIXED
- **Found:** User testing, 2026-07-03
- **Cause:** `/api/swap-recipe` in `deployment/flask_app.py` wrote the updated menu only to the old flat JSON file (`menu_file()`), but `load_menu()` reads menus from the household's database row (`load_weekly_menu_from_db`) once a household exists — which is the normal case for every real user. The write went to a file nothing reads anymore, so the change silently vanished; the "success" message was accurate about the write happening, just not about it having any effect.
- **Fix (2026-07-03):** Swap now saves to the database (with a proper open-session-and-commit, matching the pattern used elsewhere) when a household exists, only falling back to the file for the pre-household migration case.
- **File:** `deployment/flask_app.py`
- **Status:** needs re-test
- **Related — found during investigation, not yet fixed:** several other places call `append_activity_to_db(household, ...)` using `household = current_household()`, which returns an already-detached ORM object from a closed session — mutating it doesn't persist anything since nothing ever commits it. This likely means some activity log entries are silently lost (possibly why "Activity log shows correct entries after actions" was never checked off in the testing checklist). Worth an audit of every `current_household()` call site (`deployment/flask_app.py` lines ~610, 629, 648, 945, 1004, 1018) to confirm which ones need their own session+commit like the swap-recipe fix got.

### B22. Settings menu still cuts off content at the bottom on short screens — FIXED (real fix this time)
- **Found:** User testing, 2026-07-03 (after B18's first fix attempt)
- **Cause:** B18's fix used a fixed CSS `max-height: calc(100vh - 90px)`, which assumes the cogwheel button always sits ~90px from the top of the screen. On different themes/viewports the button can sit lower, so the dropdown's actual bottom edge still rendered past the visible viewport with no way to reach it — the internal scroll worked, but the box itself extended off-screen.
- **Fix (2026-07-03):** `toggleSettingsMenu()` in `frontend/static/app.js` now measures the button's actual position with `getBoundingClientRect()` when the menu opens and sets `max-height` dynamically to whatever space is really left in the viewport, so the dropdown always fits regardless of theme or screen size.
- **File:** `frontend/static/app.js`
- **Status:** needs re-test

### B24. Settings → Manage Categories list empty (only "Add category" shows) — FIXED
- **Found:** User testing, 2026-07-03
- **Cause:** `load_categories_from_db()` in `core/household_paths.py` returned `[]` immediately whenever `household.categories` was `None` in the database — which skipped the "self-heal" logic right below it that merges in the base category list from `data/categories.json`. Any household whose `categories` column was never seeded got a permanently empty list (except whatever they manually added themselves, which is why "Add" worked but the list stayed empty).
- **Fix (2026-07-03):** Removed the early return; `None` now falls through to an empty list that still gets merged with the base categories like it should.
- **File:** `core/household_paths.py`
- **Status:** needs re-test
