## ✅ ENGINEERING SECURITY PASS (2026-07-05) — fixed tonight

Prompted by "engineering:code-review" going over the code surrounding this morning's B50 fix, to catch anything else in the same shape before public launch.

- **Missing role checks on write endpoints:** `/api/add-recipe`, `/api/delete-recipe`, `/api/edit-recipe`, `/api/swap-recipe`, `/api/pantry/add`, `/api/pantry/remove`, `/api/pantry/reset`, `/api/recipe-packs/import`, `/api/recipe-packs/remove`, `/api/recipes/import` all skipped the `acting_role_can_edit()` gate already used correctly by `/api/regenerate` and `/api/categories/*` - a viewer-role profile (e.g. a kid's account) could edit the menu, delete recipes, and modify pantry/recipe-pack data. - **B50-shaped raw session reads:** ~17 call sites across `flask_app.py` read `session.get('current_household_id')` directly instead of the `current_household_id()` helper that re-validates membership against the *current* user (the actual B50 fix). Most damning: `household_settings()` fetched and rendered a household straight from the raw session value with zero re-check before display - structurally the same bug as B50 itself, just not yet triggered. All replaced with the helper.
- **No CSRF protection anywhere:** added `flask-wtf`'s `CSRFProtect`. Traditional forms (`login.html`, `signup.html`, `household-settings.html`, `admin.html`, etc.) now carry a hidden `csrf_token` field; all `fetch()`-based JSON requests get an `X-CSRFToken` header automatically via a wrapper added to `app.js`, so existing fetch calls across the codebase didn't need individual edits. Verified: an unauthenticated POST without the token now correctly gets rejected ("The CSRF token is missing").
- **PIN brute-force:** the 4-digit owner PIN (10,000 combinations) had no rate-limiting. Added an in-memory lockout in `core/auth_helpers.py` (5 wrong attempts -> 5 minute lockout, per-process - not shared across gunicorn workers or persisted across restarts, which is an acceptable tradeoff at this scale but worth revisiting with Redis if this ever runs multi-worker under real attack traffic).
- **Removed `/api/debug-token`:** left-over debugging route that returned any logged-in user's decoded Azure JWT claims; its own docstring said to remove it.

Full test suite (50 tests) still passes; app boots and renders cleanly with all changes applied.

### B50. CRITICAL: logging in as a second account on the same browser silently inherited the first account's entire household — FIXED (2026-07-05, found during multi-user testing)
- **Found while testing:** with testuser2@example.com/testuser3@example.com set up for multi-user testing, logging in as testuser2 on the same browser used for testuser1 showed testuser1's actual "Test Household" data (menu, recipes, pantry) - and writes made as testuser2 (a pantry item) actually landed in testuser1's real household record.
- **Cause:** `current_household_id()` in `deployment/flask_app.py` stores the active household in the session cookie (`session['current_household_id']`) and, once set, trusted it unconditionally on every later request - it never checked that the *currently logged-in* user was actually a member of that household. `login_local()` set `user_id`/`user_email` on login but never cleared this key, so a second login in the same browser reused the first account's still-set household id verbatim. In effect, any account could end up transparently reading and writing another account's entire household just by logging in afterward on the same browser/session.
- **Fix:** `current_household_id()` now cross-checks the session's stored household id against `get_user_households(user_id)` for the *current* user and discards it if it doesn't belong to them, falling back to one of their own households (or `None`). `login_local()` also now clears `current_household_id` on every login as a second, independent layer of defense.
- **Verified (2026-07-05):** logged in as testuser1, then testuser2 (no household of their own) in the same browser - testuser2 now correctly gets redirected to "create a household to get started" instead of seeing testuser1's data; re-logging in as testuser1 confirms their data is untouched and intact. A pantry item accidentally written into testuser1's real household during the investigation was identified and removed.

### B45. Shopping list still shows English units in Norwegian mode, contradicting B15/B20's fix — FIXED (2026-07-05, two separate causes)
- **Steps:** Switch to Norwegian → open Shopping List
- **Expected:** Per B15/B20 (already marked fixed/confirmed), units should show as "ss"/"ts"/"stk"
- **Actual (before fix):** Still showed English "tsp"/"tbsp"/"bunch" (e.g. "5 tsp", "45 tbsp", "1 bunch") in Norwegian mode
- **Cause #1 (import-time, addressed first):** `normalize_no_unit()` (the fix from B15/B20) was only ever invoked from a manual, admin-only, one-off maintenance route (`/admin/normalize-recipe-units`). The regular recipe-pack import path never called it. Fixed by having `api_recipe_packs_import()` call `normalize_no_unit()` at import time — but re-testing the *actual* shopping list afterward showed the English units **still appeared**, even though the underlying recipe data's `unit.no` field was already correctly normalized (e.g. `{"en": "tbsp", "no": "ss"}`). That meant the real bug was elsewhere.
- **Cause #2 (the real bug, found during the full regression pass):** `/shopping`'s Norwegian-mode rebuild in `deployment/flask_app.py` had its own **separate, much smaller, hardcoded unit map** (`unit_translations_no = {'pieces': 'stk', 'piece': 'stk', 'pcs': 'stk'}`) instead of reusing the shared `UNIT_MAP_NO`/`normalize_no_unit()` — so it only ever translated "pieces/piece/pcs" and silently left tbsp/tsp/bunch/etc. in English no matter how clean the underlying recipe data was. This is exactly the "different code path" the original bug note speculated about.
- **Fix:** Replaced the shopping list route's local `unit_translations_no` map with a call to the shared `normalize_no_unit()`. Also added a few missing entries to `UNIT_MAP_NO` in `core/ingredient_deduplicator.py` discovered while re-testing: `'to taste'`, `'tablespoons'`/`'teaspoons'` (plural forms), and `'clove'`/`'cloves'` → `'fedd'`.
- **Verified (2026-07-05):** Norwegian shopping list now shows "1 bunt" (was "1 bunch"), "45 ss" (was "45 tbsp"), "3 etter smak" (was "3 to taste"), "5 fedd" (was "5 cloves") — confirmed across two different generated menus/shopping lists.

### B35. Dashboard kept showing the old recipe after swapping onto a day with a brand-new recipe — FIXED (not the service worker, not sample-vs-imported recipes)
- **Found:** User testing, 2026-07-05 — swapping a *new* recipe (not already in this week's menu) onto a day updated the All Recipes sidebar correctly, but the main dashboard kept showing the old recipe
- **Investigated and ruled out:** not related to using sample vs. imported-pack recipes (recipe source doesn't affect menu state), and not the B25 service worker issue (confirmed by reading the dashboard route's own code, not by re-testing caching)
- **Real cause:** `/api/swap-recipe`'s plain-insert branch (used when the recipe isn't already planned for a different day this week - see B31) only ever updated `recipe_id`/`title`/`time_minutes`/`difficulty` on the target day, leaving `title_no`/`title_en`/`subtitle_no`/`subtitle_en`/`protein`/`image_url` stale from whatever recipe used to be there. The dashboard route (`deployment/flask_app.py:dashboard()`) resolves the displayed title by preferring `title_en`/`title_no` over the raw `title` field — so it kept showing the *old* recipe's name, since only `title` (deprioritized in that lookup) had actually been updated. Also a latent bug: `recipe['title']` is a bilingual dict for most pack recipes, so `target['title'] = recipe['title']` was assigning a dict where a string was expected.
- **Fix (2026-07-05):** The insert branch now derives and sets every display field the same way `MenuGenerator.generate_menu()` does when first building the menu (title_no/en, subtitle_no/en, protein type, image_url) — mirrored exactly, not reinvented.

- **Status:** needs re-test — swap a recipe that ISN'T already in this week's menu onto some day, confirm the dashboard immediately shows the new recipe's correct name/image

### B34. Back button restored the settings dropdown open, and sometimes re-focused a PIN field — FIXED

### B31. "Change day" silently duplicated a recipe and discarded the target day's original — FIXED
- **Found:** User testing, 2026-07-05 — a user tried to swap Wednesday's dinner with Tuesday's from inside the recipe page, expecting the two days to trade places
- **Cause:** `/api/swap-recipe` only ever inserted the recipe being viewed into the chosen target day, overwriting whatever was there — it never touched the day the recipe was originally on.
