### B35. Dashboard kept showing the old recipe after swapping onto a day with a brand-new recipe — FIXED (not the service worker, not sample-vs-imported recipes)
- **Found:** User testing, 2026-07-05 — swapping a *new* recipe (not already in this week's menu) onto a day updated the All Recipes sidebar correctly, but the main dashboard kept showing the old recipe
- **Investigated and ruled out:** not related to using sample vs. imported-pack recipes (recipe source doesn't affect menu state), and not the B25 service worker issue (confirmed by reading the dashboard route's own code, not by re-testing caching)
- **Real cause:** `/api/swap-recipe`'s plain-insert branch (used when the recipe isn't already planned for a different day this week - see B31) only ever updated `recipe_id`/`title`/`time_minutes`/`difficulty` on the target day, leaving `title_no`/`title_en`/`subtitle_no`/`subtitle_en`/`protein`/`image_url` stale from whatever recipe used to be there. The dashboard route (`deployment/flask_app.py:dashboard()`) resolves the displayed title by preferring `title_en`/`title_no` over the raw `title` field — so it kept showing the *old* recipe's name, since only `title` (deprioritized in that lookup) had actually been updated. Also a latent bug: `recipe['title']` is a bilingual dict for most pack recipes, so `target['title'] = recipe['title']` was assigning a dict where a string was expected.
- **Fix (2026-07-05):** The insert branch now derives and sets every display field the same way `MenuGenerator.generate_menu()` does when first building the menu (title_no/en, subtitle_no/en, protein type, image_url) — mirrored exactly, not reinvented.
- **File:** `deployment/flask_app.py`
- **Status:** needs re-test — swap a recipe that ISN'T already in this week's menu onto some day, confirm the dashboard immediately shows the new recipe's correct name/image

### B34. Back button restored the settings dropdown open, and sometimes re-focused a PIN field — FIXED
- **Found:** User testing, 2026-07-05 — using the new Back button (B33) from Household/Settings (opened via the cogwheel dropdown) reopened the dropdown itself; separately, going into Settings sometimes auto-focused the PIN input again (popping the mobile keyboard) despite B19's earlier fix.
- **Cause:** Both are the same root cause — browser bfcache (back/forward cache) restores a page exactly as it looked the moment the user navigated away, including UI state like an open dropdown or a focused input. If the settings dropdown was open (or a PIN field was focused) right before clicking a link out of it, hitting Back brought that exact state back.
- **Fix (2026-07-05):** Added a `pageshow` listener in `base.html` that closes the settings dropdown/theme submenu and blurs whatever input has focus whenever a page is restored from bfcache.
- **File:** `frontend/templates/base.html`
- **Status:** needs re-test

### B31. "Change day" silently duplicated a recipe and discarded the target day's original — FIXED
- **Found:** User testing, 2026-07-05 — a user tried to swap Wednesday's dinner with Tuesday's from inside the recipe page, expecting the two days to trade places
- **Cause:** `/api/swap-recipe` only ever inserted the recipe being viewed into the chosen target day, overwriting whatever was there — it never touched the day the recipe was originally on. So "moving" a recipe from Wednesday to Tuesday actually left it on both days, while Tuesday's original recipe just vanished with no warning.
- **Fix (2026-07-05):** Now a true swap — if the recipe is already planned for a different day this week, the two days trade every field (recipe, times, difficulty, protein, etc.), so nothing is lost or duplicated. If the recipe isn't currently in this week's menu at all (e.g. inserting a brand-new one from the catalog), it's still a straight insert into the target day, since there's nothing to swap it with.
- **Files:** `deployment/flask_app.py`, `frontend/templates/all-recipes.html`, `frontend/templates/recipe.html`
- **Status:** needs re-test — from a recipe's page, pick a day that already has a different recipe, confirm both days trade places instead of duplicating

### B29. Settings dropdown renders off-screen when the cogwheel sits near the left edge — FIXED
- **Found:** User testing, 2026-07-05 — happens when the browser's unit/zoom scaling shifts the nav so the cogwheel lands on the left side of the screen
- **Cause:** `.settings-dropdown` is right-aligned to its button (`right: 0`) and expands leftward — fine when the button sits on the right side of the nav (the common case), but if the button ends up near the left edge, that leftward expansion runs off the left of the screen with no way to reach those items.
- **Fix (2026-07-05):** `toggleSettingsMenu()` in `frontend/static/app.js` now checks the dropdown's actual rendered position on open and clamps it to `left: 8px` if it would otherwise overflow off the left edge, same approach as the earlier vertical (B22) fix.
- **File:** `frontend/static/app.js`
- **Status:** needs re-test
