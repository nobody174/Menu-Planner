## 🚨 LAUNCH BLOCKERS — must resolve before public + paid launch (do NOT skip)

Flagged 2026-07-05 during a legal/compliance check ahead of the planned public launch + paid Patreon tier (99 kr/mo, 7-10 day trial). None of this blocks the current friends-and-family soft launch — this is specifically gating the move to (a) truly public signups and (b) turning on the paywall.

### L1. No privacy policy or terms of service exist anywhere (repo or live site)
- Required before any paid or public launch. You're the data controller (Norway/EEA, GDPR applies); Neon + Resend are both EU-hosted per your confirmation, which simplifies this.
- Needs to cover: what data is collected (email, password hash, household member names/avatars, referral codes, activity log, pantry/recipe data), why (contract/legitimate interest for core functionality), how to request deletion (Danger Zone already covers this) or access, and that household members' data is entered by the account owner.
- Claude offered to draft both in plain language — not done yet as of this writing.

### L2. Trial-to-paid conversion needs an explicit pre-charge notice
- Norway's Forbrukertilsynet actively enforces against subscriptions that silently convert a free/trial period into a paid one (negative option billing, markedsføringsloven §11).
- Needs: a clear email/in-app reminder before the 7-10 day trial converts to the 99 kr/month charge, not just a UI countdown.

### L3. Cancellation must be self-serve and as easy as signup
- Not yet built (no paywall exists yet). When built, cancellation must live in Settings, not "email us to cancel."

### L4. 14-day right of withdrawal (angrerettloven) for EEA consumers
- Applies to the paid subscription once launched. Needs clear disclosure at signup/payment.

### L5. Tax / business registration timing undecided
- Currently running as an individual (no enkeltpersonforetak). Not a blocker for friends-and-family, but recurring subscription income is taxable from day one regardless of registration status. Decide a real trigger point (e.g. before X paying users or €X/month) rather than leaving it open-ended.

### L6. Confirm Patreon's own creator/refund terms don't conflict with your stated "99 kr/month" terms
- Patreon acts as payment processor and imposes its own refund/payout rules on you as a creator — read these once before publishing your own subscription terms.

**Also discussed, not yet decided:** whether Patreon is the right payment path at all vs. Stripe or another processor (Lemon Squeezy/Paddle also discussed as merchant-of-record options that absorb EU VAT complexity) — worth revisiting once ready to build the actual paywall.

## QA / PRODUCTION-READINESS REVIEW FINDINGS (2026-07-06)

Full code + live-functional review (see CHANGELOG 2026-07-06 for the session summary). Nothing here is release-blocking on its own; see notes per item.

### High Priority

**B53. Silent partial/degraded menu generation**
- `core/menu_generator.py:434-438`, `deployment/flask_app.py:3018` (`/api/regenerate`). If a household has fewer eligible recipes than the requested day count, the endpoint returns `200 success` with a shorter menu instead of telling the user why. Fix: return a clear message ("only 3 recipes matched your selected categories") instead of a silent shortfall.

**B54. CI's dependency vulnerability scan doesn't gate anything**
- `.github/workflows/ci.yml:133` runs `safety check || true` - any CVE finding is informational only, never fails the build. Combined with exact `==` pins on early-2024-vintage packages (Flask 3.0.0, Werkzeug 3.0.1, requests 2.31.0), there's no automated catch for a newly disclosed CVE in an already-pinned version. Fix: drop `|| true`, or alert on findings.

**B55. Missing "no results" empty state on All Recipes search**
- Verified live: searching for a nonsense term (e.g. "zzzznonexistent") correctly hides every recipe card, but the page goes blank under the search box with no "No recipes found" message, and the heading still reads "All Recipes (277)" instead of the filtered count. Reads as broken on first encounter. Low effort fix.

### Medium Priority

**B56. Missing DB indexes on frequently-queried columns**
- `households.owner_id` and `household_members.household_id`/`user_id` (`database/models.py` ~82, 124-126) are queried on nearly every request but have no index. Not urgent at current scale, but a real cost as household count grows.

**B57. `deployment/flask_app.py` has grown to ~4,000 lines / ~70 routes**

**B58. Firefox-only rendering bug: white block at bottom of page**

Not yet reproduced or diagnosed - this environment's Chrome-only browser automation (Claude in Chrome) can't reach a real Firefox engine, and an attempt to install Playwright's bundled Firefox/WebKit binaries in the sandbox failed (outbound download blocked by the sandbox's network allowlist: `cdn.playwright.dev` returns 403). Needs either: (a) the user to note which specific page(s) this happens on plus a screenshot, so the CSS can be diagnosed from the code directly, or (b) a CI job added that runs Playwright's Firefox engine automatically on every push (Playwright downloads fine from GitHub Actions runners, which aren't behind this sandbox's allowlist) - see note below on cross-browser testing strategy.

### Cross-browser / cross-device testing strategy (see B58) — RESEARCHED & DECIDED 2026-07-06

Two research passes (via Claude subagents) confirmed a concrete, fully free plan. Real QA teams don't install one of every browser locally either - this is the same approach they use.

**B62. Set up Playwright MCP + Playwright CI suite (not yet done - next step for Claude Code)**

1. **Playwright MCP, for live interactive checks during development** - confirmed real, maintained by Microsoft (`microsoft/playwright-mcp`). Install once via `claude mcp add playwright npx @playwright/mcp@latest` (Node 20+ required). Gives Claude Code the ability to drive a real Chromium/Firefox/WebKit browser directly from a plain-English instruction ("check this page in Firefox," "show me the iPhone 14 view") - no more manual DevTools device swapping. Confirmed it supports `--browser=firefox`/`webkit` and named device presets (viewport+UA+touch auto-set) via a resize/emulate tool.
2. **Playwright test suite in CI, for automatic regression coverage on every push** - `npm init playwright@latest` in the repo, point `baseURL` at `http://localhost:5000`, define projects per browser (chromium/firefox/webkit, all three via Playwright's own bundled engines) and per device (`playwright.devices` presets - iPhone SE, iPhone 14, iPad, Pixel 7, etc., no manual preset picking needed). Add `await expect(page).toHaveScreenshot()` on key pages (dashboard, all-recipes, shopping, add-recipe) for pixel-diff visual regression - this specifically would have caught the Firefox white-block bug (B58) automatically, before a human ever saw it. Wire as a new job in the existing `.github/workflows/ci.yml`, running on every push. Confirmed free: GitHub Actions minutes are free and unlimited on a public repo (2026 pricing verified), Playwright's snapshot diffing is self-hosted/free with no third-party account.
3. **Cloud device farms (BrowserStack/Sauce Labs/LambdaTest): skip these.** Verified current free tiers are thin or time-limited trials now (BrowserStack: 30-100 min trial that expires; LambdaTest: permanent free plan but capped at 1 parallel test / 10 checks per month). Playwright's built-in device emulation already covers the viewport/UA/touch bug class described here; paid device farms only add value for genuine hardware-specific quirks, not this.

**Also add to CLAUDE.md** once set up: an instruction that after any change to a template/HTML/CSS file, Claude Code should run the Playwright visual-check suite (or drive Playwright MCP directly) before considering a UI change complete - this is a workflow rule, not a one-time task, so it belongs in CLAUDE.md alongside the existing push/deploy instructions.
