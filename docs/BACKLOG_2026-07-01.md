**B63. SQLite concurrency bug — RESOLVED 2026-07-07, live-affected real usage (not just tests)**
- First surfaced 2026-07-06 while building the Playwright suite (`IndexError: tuple index out of range` under parallel test workers) and was assumed to be a test-only artifact, fixed there by pinning Playwright to `workers: 1`.
- 2026-07-07: the same underlying issue turned out to genuinely affect normal live usage against the local SQLite dev database - the 🎲 reroll button and the recipe-search "swap" button both intermittently failed with a generic client-side error (the frontend's error handling swallows the real server message on a non-JSON/500 response, so the user only ever saw a blank error dialog). Root cause: SQLAlchemy's `StaticPool` forces every thread onto one shared raw sqlite3 connection; Flask's dev server threads by default, so two overlapping requests (a real browser's multiple simultaneous connections per page load, not just a double-click) corrupted that connection's cursor state mid-query - confirmed via `sqlite3.InterfaceError`, `sqlite3.OperationalError`, and `IndexError`, all symptoms of concurrent access to one connection, not an application logic bug.
- Two fix attempts made things worse before landing on the right one - noted here so nobody re-tries them: (1) a global lock around connection checkout/checkin deadlocked the *entire app* (including totally unrelated actions like Settings and Generate New Menu) if any single request died mid-flight without releasing the lock; (2) `threaded=False` on the Flask dev server avoided the corruption but choked on a real browser's normal multiple-connections-per-page-load behavior, so the app appeared to hang on unrelated actions again.
- Actual fix (`database/database.py`): enabled SQLite's own WAL (Write-Ahead Logging) journal mode plus a 5s `busy_timeout`, and switched off `StaticPool` back to a normal per-thread connection pool (WAL only delivers its concurrent-access safety with separate connections per thread - `StaticPool` was defeating it). This is the standard, well-established fix for Flask + SQLite concurrent access - no custom locking, no threading restriction. Verified: 24 concurrent reroll requests via Flask's real threaded dev server, 0 failures (previously failed reliably under the same load); full 50-test pytest suite still passes.
- Production is unaffected either way - it runs gunicorn workers against Postgres, which already handles real concurrency safely.

Status: researched and decided, not yet implemented. Concrete Claude Code prompt to kick this off is saved below in CHANGELOG.md's 2026-07-06 entry (or ask Claude again for it - it was also given directly in chat).

**B65. All Recipes page never collapses to a mobile layout - sidebar+content overflows off-screen — RESOLVED 2026-07-07**
- Found 2026-07-07 while manually inspecting the Playwright suite's per-device screenshots for an unrelated change (B53) - not something a pixel-diff catches on its own, since there was never a passing baseline to diff against on this machine; only caught by actually opening the images. Confirmed unrelated to B53 - `frontend/templates/all-recipes.html` had zero changes in that diff, and this reproduced identically on `main` before B53 too.

**B67. Auto-tagged versions never got a GitHub Release — RESOLVED 2026-07-07**
- Found 2026-07-07 by the owner asking why v1.1.1/v1.1.2 didn't show up as Releases on GitHub, only v1.1.0 (from before auto-tagging existed). `git ls-remote --tags` confirmed the tags genuinely existed on GitHub; `gh run list --workflow=release.yml` showed **zero runs ever fired from a tag push** in this repo's entire history, despite `release.yml` having an `on: push: tags: 'v*'` trigger that should have caught every auto-tag since it was introduced.
- Root cause: the auto-tag step in `ci.yml`'s deploy job pushes using the workflow's own `GITHUB_TOKEN` (implicitly, via the checkout action's stored credentials). GitHub deliberately blocks any push authored by `GITHUB_TOKEN` from triggering other workflows - a documented anti-infinite-loop protection. So the tag push succeeded and the tag was real, but it was invisible to `release.yml`'s trigger the entire time.
- Fix: create the GitHub Release directly inside the same `ci.yml` deploy job (`gh release create`, right after the auto-tag step) instead of relying on a second workflow to pick up the tag push. Sidesteps the cross-workflow trigger restriction entirely rather than working around it (e.g. with a PAT). `release.yml` itself is left as-is and still works correctly for a manually-pushed tag (`git tag vX.Y.0 && git push` from a real user's local git) - only auto-tagged CI pushes were ever affected.
- v1.1.1 and v1.1.2 remain tag-only with no Release (not retroactively fixed - low value, they're already superseded by newer work); this fix takes effect starting with the next auto-tagged deploy.

**B68. Playwright e2e seed data was non-deterministic, causing real CI flakiness — RESOLVED 2026-07-07**
- Found 2026-07-07: a push to `main` containing only workflow/doc changes (B67, `.github/workflows/ci.yml` + docs, no template/CSS/JS touched at all) still failed "Stage 2: Visual Regression (Playwright)" on `dashboard-iPhone-SE`, 3/3 attempts (2 retries). The diff image showed genuinely different recipes on Monday/Tuesday/Wednesday between the baseline and this run ("Fish Cakes with Remoulade" vs "Grilled Salmon with Asparagus", etc.) - not a layout or rendering difference at all.
- Root cause: `e2e/seed_test_data.py` calls `MenuGenerator(household_id=household_id)` with no `seed=` argument. `MenuGenerator.__init__` only calls `random.seed(seed)` `if seed:` - so every seed run (every CI run, every local run) generates a genuinely different random weekly menu. Any committed baseline screenshot of the dashboard or All Recipes page can only ever match a run that happens to randomly reproduce the same day-by-recipe assignments - this was always going to eventually flake, it just hadn't yet.
- Fix: pass a fixed `seed=42` to `MenuGenerator` in `e2e/seed_test_data.py`. Verified locally: two independent seed runs now produce byte-identical Monday/Tuesday/Wednesday recipe assignments.
- This changes the actual seeded menu content, so every existing `dashboard-*` and `all-recipes-*` baseline needed regenerating again (same process as B53/B65's baseline update - pull the real `-actual.png` files from a CI run and commit them as the new baselines, since Linux baselines can't be generated locally on this Windows machine).
