## 1. Executive Summary

This is a healthy, unusually well-documented solo project. The backlog discipline
(B-IDs, root-cause write-ups, verified fixes), staged CI with real gates,
7-environment visual regression, and honest CLAUDE.md workflow notes are better than
most small commercial codebases. Recent fixes (B53–B68) check out — no evidence was
found contradicting the backlog's stated statuses.

The material risks cluster in four areas:

- **(a)** a handful of genuinely new production-facing bugs (a likely-broken
  shopping-list export in production, a broken `/themes` route, a session
  secret-key fallback that plausibly connects to B17),
- **(b)** auth hardening gaps (no login rate limiting, account enumeration),
- **(c)** the storage-layer split that B61 already tracks but which is wider than
  B61 states (it's a *triple* path, with schema drift and dead columns), and
- **(d)** deployment definition split-brain (Railway-era Docker vs. Render Procfile
  with contradictory worker counts and assumptions baked into code comments).

Nothing found blocks the current friends-and-family soft launch. Several items
should join L1–L6 as gates before truly public signups.

## 2. Architecture Overview

- **Stack:** Flask 3.1 + SQLAlchemy 2 + Alembic, Jinja templates with vanilla JS
  (no framework, no bundler), PWA manifest + service worker. Postgres (Neon) in
  production, SQLite+WAL for local dev. Resend for transactional email. Render
  behind Cloudflare, deployed via a deploy hook fired from GitHub Actions Stage 3.
- **Data model:** Users → Households → Members/Profiles (Netflix-style). Household
  content (recipes, menu, pantry, categories, activity log) lives in **JSONB
  columns on `households`** — the relational `recipes`/`weekly_menus`/
  `shopping_lists` tables still exist but are effectively legacy, as is a parallel
  per-household flat-file tree under `data/households/` (B61).
- **App shape:** one 4,484-line `deployment/flask_app.py` with ~80 routes organized
  by comment banners (B57), hand-rolled session auth (no flask-login despite it
  being pinned in requirements), CSRFProtect global, role checks via
  `acting_role_can_edit()`/`acting_role_is_owner()`, and row-level locking
  (`locked_household()`) for menu mutations.
- **CI/CD:** 3-stage pipeline (lint/data-validation/frontend → tests/security/
  build/docker/playwright → deploy+tag+release), branch protection on
  `public-release-v1`.

## 3. Strengths

- **Security fundamentals mostly right:** pbkdf2 password hashing, CSRF everywhere
  (including a fetch wrapper injecting `X-CSRFToken`), secure/httponly/samesite
  cookies, enumeration-safe forgot-password, one-time expiring reset tokens, PIN
  brute-force lockout, the cross-account household-leak guard in
  `current_household_id()` (flask_app.py:788-800), and `send_from_directory`
  (traversal-safe) for file serving.
- **Frontend XSS discipline:** every `innerHTML` sink sampled either writes static
  markup or escapes user data via `_esc()`/DOM-node construction — with comments
  showing this was a deliberate, learned pattern.
- **Operational honesty in code:** comments document *why* (WAL fix,
  `flag_modified` root cause, gosu signal handling, GITHUB_TOKEN anti-loop) rather
  than *what*. `conftest.py` refuses to run against the real dev DB — a genuinely
  thoughtful guard.
- **CI is real:** pip-audit gates the build (B54 fix confirmed in ci.yml:141),
  Playwright visual regression is a required check, deploys verify live `/health`
  before tagging.
- **Documentation is current:** SYSTEM_ARCHITECTURE.md matches the actual
  Render/Neon/Resend reality (updated 2026-07-02); the backlog file is trustworthy
  — everything it claimed fixed, the audit found fixed.

## 4. Findings

### Critical

**C1. Shopping-list export/sync likely broken in production (bare
`shopping_integrations` import)** — *New*
- **Description:** `/api/export-shopping-list` (flask_app.py:3187) and
  `/api/sync-shopping-list` do `from shopping_integrations import ...` — a bare
  module import. The only `sys.path` insert (flask_app.py:384) adds the **project
  root**, not `deployment/`. When the app runs as
  `gunicorn deployment.flask_app:app` (the Procfile), `deployment/` is a package,
  so a bare `shopping_integrations` cannot resolve — the route's `try/except`
  catches the `ImportError` and returns a 500. It works locally only when
  `flask_app.py` is executed directly (script dir gets auto-added to path).
- **Why it matters:** A user-facing feature silently 500s in production while all
  CI checks pass (imports are lazy inside the route; no route-level tests exist —
  B59).
- **Affected files:** `deployment/flask_app.py` (~3187, ~3263),
  `deployment/shopping_integrations.py`.
- **Suggested improvement:** Change to
  `from deployment.shopping_integrations import ...`. Then click-test export live.
- **Confidence:** High on the mechanism; Medium that it manifests on Render
  (depends on Render's exact CWD/path setup — a one-minute live test settles it).
  If export demonstrably works live, downgrade to a fragility finding.

### High

**H1. `SECRET_KEY` silently falls back to a random per-process value — plausible
B17 contributor** — *New; cross-references B17*
- **Description:** flask_app.py:515:
  `os.environ.get("FLASK_SECRET_KEY", secrets.token_hex(32))`. If the env var is
  missing, every restart (and, under the Docker entrypoint's `--workers 4`, every
  *worker*) gets a different signing key — all existing sessions become invalid,
  users get logged out with no error.
- **Why it matters:** This is exactly B17's symptom (intermittent relogin). The
  30-day-session fix shipped 2026-07-03 helps only if the key is stable. Render's
  env vars aren't visible from the repo, so it can't be confirmed set — but the
  *fallback itself* is the defect: a missing production secret should crash at
  boot, not degrade invisibly.
- **Affected files:** `deployment/flask_app.py:515`.
- **Suggested improvement:**
  `if IS_PRODUCTION and not os.environ.get("FLASK_SECRET_KEY"): raise RuntimeError(...)`.
  Separately, confirm the var is set on Render — if it wasn't, this may be B17's
  real root cause.
- **Confidence:** High (code), Medium (B17 linkage).

**H2. No rate limiting or lockout on password login, signup, or password-reset
requests** — *New*
- **Description:** `/login` (flask_app.py:2568) accepts unlimited attempts. The
  2026-07-05 security pass added a lockout for the 4-digit PIN but left the primary
  credential endpoint unprotected. `/forgot-password` and `/signup` can also be
  scripted freely (email-sending abuse, referral-code farming).
- **Why it matters:** Online brute-force against real accounts is the most common
  credential attack; before public signups this becomes internet-facing.
- **Affected files:** `deployment/flask_app.py` (login/signup/forgot-password
  routes), `core/auth_helpers.py`.
- **Suggested improvement:** flask-limiter (in-process is fine at one worker), or
  reuse the PIN-tracker pattern keyed by email+IP on failed logins. Not needed for
  friends-and-family; should gate public launch.
- **Confidence:** High.

**H3. `/themes` gallery route crashes with 500** — *New*
- **Description:** `theme_gallery()` (flask_app.py:4023) lists
  `frontend/static/theme-previews/` — **that directory does not exist** (verified;
  the real one is `frontend/static/themes/previews/`, which holds CSS, not the HTML
  files the route expects). `os.listdir` raises `FileNotFoundError` → 500.
  `/themes/<filename>` is equally dead.
- **Why it matters:** Anyone hitting the URL gets the Oops page; it's also dead
  weight from the theme refactor era.
- **Affected files:** `deployment/flask_app.py:4023-4047`,
  `frontend/templates/theme_gallery.html`.
- **Suggested improvement:** Delete both routes and `theme_gallery.html` unless a
  linked entry point still exists (none found in the nav — only
  `theme_gallery.html` itself links there).
- **Confidence:** High (directory absence verified).

### Medium

**M1. Account deletion can fail or leave dangling references** — *New; relates to
L1 (GDPR deletion promise)*
- **Description:** `delete_user_account()` (core/auth_helpers.py:282) deletes
  memberships → households → user via `query.delete()`, which **bypasses ORM
  cascades**. Rows in the legacy relational tables (`recipes.household_id`/
  `created_by`, `weekly_menus`, `shopping_lists`) would trigger Postgres FK
  violations → the whole deletion rolls back and Settings shows "Database error".
  Referees' `referred_by_user_id` is never nulled (no DB-level FK exists in prod —
  see M2 — so no error, but a dangling user id persists after a GDPR erasure).
- **Why it matters:** L1's privacy policy will promise working self-serve deletion
  ("Danger Zone already covers this") — this path has failure modes exactly for
  users with legacy rows or referral relationships.
- **Affected files:** `core/auth_helpers.py:282-317`, `database/models.py`.
- **Suggested improvement:** Null out `referred_by_user_id` for referees, and
  either delete legacy child rows explicitly or confirm the relational tables are
  empty in production and drop them (folds into B61's cleanup).
- **Confidence:** High on mechanism; Medium on real-world trigger (depends on
  whether legacy rows exist in prod).

**M2. Model/migration schema drift; dead `removed_categories` JSONB column** —
*New; extends B61*
- **Description:** models.py:48 declares `ForeignKey("users.id")` on
  `referred_by_user_id`, but the migration (`b2c3d4e5f6a7`) created a plain string
  column — dev SQLite (via `create_all`) and prod Postgres (via Alembic) genuinely
  differ. Meanwhile the `households.removed_categories` JSONB column exists in the
  schema but **is never read or written** — category tombstones go only to the
  filesystem (`mark_category_removed`, flask_app.py:1801/1868), and
  `load_categories_from_db()` reads the file version too.
- **Why it matters:** Deleted-category state for DB-backed households lives on the
  Render disk, not in Neon. Any disk loss/detach/migration (or a DB-only restore)
  silently resurrects every category a household deleted, because the JSONB
  self-heal re-adds base categories with no surviving tombstone. It's also exactly
  the dual-path trap B61 describes, in a place B61 doesn't list.
- **Affected files:** `database/models.py:48,98`, `core/household_paths.py`
  (tombstone functions, `load_categories_from_db`),
  `alembic/versions/b2c3d4e5f6a7_*.py`.
- **Suggested improvement:** Move tombstones into the existing JSONB column (it's
  already there waiting); align model and migration for the referral FK.
- **Confidence:** High (code paths verified); Medium on production impact (depends
  on the persistent disk never being lost — which is the point).

**M3. Deployment definition split-brain: Railway-era Docker vs. Render Procfile** —
*New*
- **Description:** Two contradictory production definitions coexist:
  `docker-entrypoint.sh` (Railway comments, **4 gunicorn workers**, runs Alembic
  migrations, seed-restore logic) vs. `Procfile` (Render path, default **1
  worker**, no migration step). Code comments encode single-worker assumptions
  ("it doesn't today" in `locked_household()`, per-process PIN lockout, in-memory
  rate limiter) that the Docker path violates. It's also unclear from the repo
  which one Render actually executes and therefore who runs migrations in
  production.
- **Why it matters:** The next person (or session) to touch scaling or deployment
  will trust one of these and be wrong. If Render ever moves to Docker or multiple
  workers, the PIN lockout quietly weakens 4× and H1 becomes a live outage.
- **Affected files:** `Procfile`, `Dockerfile`, `docker-entrypoint.sh`,
  `docker-compose.yml`.
- **Suggested improvement:** Decide which artifact is production, delete or clearly
  demote the other (`docs/` note), and document where migrations run on Render.
- **Confidence:** High that they diverge; the "which is live" question needs the
  Render dashboard.

**M4. Silent write-failure swallowing in storage helpers** — *New; relates to B61*
- **Description:** `save_recipes_db()` (flask_app.py:1161-1195) catches commit
  exceptions, `print()`s, and returns normally — the route then reports success on
  a lost write. `_save_pantry_db()` (flask_app.py:936) does the same and
  *additionally* falls back to writing a file that the DB-first read path will
  never look at again. Six `print()` calls sit where `logger` should be. Also:
  flask_app's own `save_recipes_db` duplicates
  `household_paths.save_recipes_db_to_db()` but **skips `flag_modified()`** — it
  currently persists only by the accident of re-querying in a fresh session, the
  precise trap the B63/swap-day comment warns about.
- **Why it matters:** "200 success, nothing saved" is this codebase's own
  documented worst bug class (B53, B63, swap-day). These helpers keep the pattern
  alive.
- **Affected files:** `deployment/flask_app.py:903-960,1161-1195`.
- **Suggested improvement:** Let exceptions propagate (or return failure), replace
  `print` with `logger`, route all JSONB saves through the `household_paths`
  `*_to_db` functions.
- **Confidence:** High.

**M5. API error responses are inconsistent; 500s return HTML to JSON clients** —
*New; extends B63's frontend observation*
- **Description:** The global 500 handler (flask_app.py:4464) renders `error.html`
  for every path including `/api/*` — which is why the frontend "swallows the real
  server message on a non-JSON response" (B63's own words). Meanwhile several
  routes return raw `str(e)` to the client (e.g. recipes import/export) — internal
  detail leakage and no stable error shape.
- **Affected files:** `deployment/flask_app.py:4459-4467` and various `/api/*`
  routes.
- **Suggested improvement:** Register a JSON error handler for `/api/` paths
  (check `request.path` in the handler); standardize on
  `{"success": bool, "message": str}` — both current shapes (`status` vs
  `success`) coexist today.
- **Confidence:** High.

**M6. Per-request performance waste (compounds B56)** — *New*
- **Description:** `current_household_id()` (46 call sites) opens a fresh session
  and runs a membership query on **every call** — a single dashboard request
  repeats the identical lookup several times. `_load_i18n()` re-reads the 29KB
  i18n.json from disk per request (docstring says deliberately). `find_recipe()`
  re-reads `sample_recipes.json` from disk per call and is called in per-dinner
  loops. On unindexed columns (B56) against Neon over the network, this stacks.
- **Affected files:** `deployment/flask_app.py:31-39,775-807,1197-1210`,
  `core/household_helpers.py`.
- **Suggested improvement:** Cache household id/role on `flask.g` per request;
  cache i18n with an mtime check in production. Do this alongside B56's indexes.
- **Confidence:** High on the pattern; impact is Low today, real at growth.

**M7. Login account enumeration** — *New*
- **Description:** `/login` renders `authenticate_user`'s literal errors: "User not
  found" vs "Invalid password" (flask_app.py:2596); `/resend-confirmation`
  distinguishes "User not found" / "Email already confirmed". (Forgot-password gets
  it right.) `authenticate_user` also returns early for unknown users without a
  hash comparison — a timing oracle.
- **Affected files:** `deployment/flask_app.py:2568-2660`,
  `core/auth_helpers.py:147-170`.
- **Suggested improvement:** Generic "Invalid email or password" for both cases; a
  dummy hash check for unknown emails. Cheap fix, do it with H2.
- **Confidence:** High. Severity kept Medium — signup necessarily reveals existence
  anyway; this mainly matters once paired with H2's unlimited attempts.

**M8. No security response headers** — *New*
- **Description:** No `Content-Security-Policy`, `X-Frame-Options`/
  `frame-ancestors`, `X-Content-Type-Options`, or `Strict-Transport-Security`
  anywhere (verified by grep; Cloudflare may add HSTS but nothing app-side).
- **Why it matters:** Clickjacking protection and a CSP safety net cost ~10 lines
  in an `after_request` hook; the app's escaping discipline is good, but CSP is the
  backstop for the sink nobody sampled.
- **Affected files:** `deployment/flask_app.py` (no `after_request` exists).
- **Suggested improvement:** Start with `X-Frame-Options: DENY`,
  `X-Content-Type-Options: nosniff`, and a report-only CSP (the inline JS/CSS
  everywhere means an enforcing CSP needs nonce work — defer that).
- **Confidence:** High.

### Low

**LO1. No request body size limit** — *New.* `MAX_CONTENT_LENGTH` unset;
`/api/recipes/import` accepts unbounded JSON (validated only for `id`/`title`
presence) straight into the household's JSONB column. Self-inflicted mostly, but a
100MB pantry POST shouldn't be possible. Set a sane cap (e.g. 2MB).
Confidence: High.

**LO2. Unauthenticated API requests get 404/empty instead of 401** — *New.*
`/api/menu`, `/api/recipes/search`, `/api/recipes/export` have no explicit auth
check; they're safe only because `current_household_id()` returns None. There is no
`@login_required` decorator anywhere — every route hand-rolls its gate, so safety
rests on every data helper resolving through household scoping. One future
non-household-scoped route that forgets a check becomes a leak. Suggest a small
decorator now, adopted incrementally (pairs naturally with B57's blueprint split).
Confidence: High (pattern), speculative (future risk).

**LO3. Dead Pi-era modules and config** — *New; extends B60.*
`deployment/scheduler.py`, `deployment/email_notifier.py`,
`deployment/to_do_sync.py` are imported nowhere (verified). Most of `config.py` is
stale (SMTP settings superseded by Resend, `SCHEDULER_ENABLED`, `HOUSEHPLD` typo,
`MENU_DAYS = 5` vs. the actual 6-day `DAYS` list in menu_generator.py:52,
placeholder "nobody174" attribution headers repo-wide). Delete or annotate.
Confidence: High.

**LO4. Alembic housekeeping** — *New.*
`alembic/versions/a1b2c3d4e5f6_add_password_reset_to_users.py` actually contains
revision `g6h7i8j9k0l1` — two files share a filename prefix and one lies about its
revision. The chain itself is valid (verified linear d0d40b4db7ac→…→f6a7b8c9d0e1);
this is purely a future-confusion trap. Rename the file. Confidence: High.

**LO5. `datetime.utcnow` throughout models/helpers** — *New.* Deprecated since
Python 3.12; the repo pins 3.11.9 so it's future debt only. Confidence: High.

**LO6. Confirmation/reset tokens stored in plaintext** — *New.* A DB leak makes
live tokens directly usable; reset tokens expire in 1h (good) but confirmation
tokens never expire. Hash tokens at rest for belt-and-braces; low priority at this
threat model. Confidence: High (fact), opinion (priority).

**LO7. `/health` doesn't touch the DB, and the deploy gate can false-pass** —
*New.* flask_app.py:4009 returns healthy unconditionally, and the CI health poll
starts immediately after firing the deploy hook — during a rolling deploy the *old*
instance answers healthy, "confirming" a deploy that hasn't finished. Add a cheap
`SELECT 1` and ideally compare a build/version identifier in `/health` against the
just-deployed SHA. Confidence: High (code), Medium (race actually occurring).

**LO8. Misc cosmetics** — *New.* `delete_own_account` re-imports
`delete_user_account` twice and does household-folder cleanup *after* the user is
deleted (`current_household_id()` is None by then — dead code);
`save_menu`/`_save_pantry_db` file fallbacks can build a `households/None/` path
when unauthenticated; `/health` leaks a `"https": CERT_FILE.exists()` field that's
meaningless on Render. Confidence: High.

## 7. Production Readiness

**Verdict: Ready for the current friends-and-family soft launch; ready for public +
paid launch only after minor-to-moderate improvements.**

Reasoning: core flows are solid, actively verified, and CI genuinely gates deploys.
But a public launch means hostile traffic and paying EU consumers, and two gaps sit
outside what L1–L6 cover:

- **L1–L6 cover the legal/compliance side well** (privacy policy, trial conversion,
  cancellation, withdrawal, tax, Patreon terms) — nothing in this audit contradicts
  or expands them, though **M1 directly underpins L1**: the policy will promise
  self-serve deletion, and that code path has real failure modes today.
- **L1–L6 do not cover technical launch-hardening.** A de-facto "L7" bundle is
  recommended as public-launch gates: H1 (secret-key fail-hard + verification), H2
  (login rate limiting), M7, M8, LO1 — plus C1 verified fixed. None is more than a
  day's work; together they're the difference between "works for people who like
  you" and "safe against people who don't".
- B17 remains open per the backlog; H1 is the most concrete new lead on it and is
  worth checking before more monitoring time is spent.
