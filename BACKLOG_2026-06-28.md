### 12. Send referral link via email/SMS directly from the app (research only, not started)
**Background:** users currently have to manually copy/paste the referral link to share it. The user asked: could the app instead take a friend's email or phone number and send the link for them — like the common "enter your email, get a copy" pattern seen on many sites? Two research passes were run; findings reconciled below (the second pass added more precision on exact free-tier limits).

**Email sending — cheap and low-risk to add soon:**
- **Resend** is the standout: 3,000 emails/month *and* 100/day, permanently free (current "New Free Tier," not a trial). Needs a verified sending domain (SPF/DKIM DNS records) to send to arbitrary recipients — a one-time setup cost that also improves deliverability for any other app email (password resets, etc.), so it's not feature-specific overhead. Official Python SDK + documented Flask integration guide exists.
- **Postmark** also has a genuinely permanent free tier, but capped hard at 100 emails/month with no overage — sending just stops once hit. Fine as a fallback, less headroom than Resend.
- **Ruled out:** Amazon SES (free tier is 12-months-only, not permanent, and sandbox mode blocks sending to unverified recipients — a dealbreaker since referral recipients are by definition unverified); SendGrid (permanent free plan discontinued as of 2025, now a 60-day trial only); Mailgun (free plan exists but limited — 1 domain, 1-day log retention); Gmail SMTP (free but throttled ~500/day with reputation-based flagging reported after as few as ~12 rapid sends, not intended for product/transactional use).
- `Flask-Mail` works via SMTP across most of these, so the integration pattern is reusable regardless of provider choice.
- **Verdict: worth doing soon.** Resend's free tier comfortably covers realistic referral volume at $0, setup is proportionate (domain DNS once + API key + a route + a template).

**SMS sending — defer until there's revenue:**
- No provider (Twilio, Vonage/Nexmo, AWS SNS/End User Messaging, Nordic specialist Link Mobility) has a meaningful free tier — every message costs money from message #1 (~$0.065/message to Norway via Twilio; AWS and Vonage broadly comparable for Nordic destinations).
- **Norway-specific landmine:** Norwegian carriers (Telenor, Telia, Ice) are documented to filter/block SMS containing URLs unless the sender ID and content are pre-allowlisted with the carrier through the provider — and the entire point of this feature is sending a clickable link. Link Mobility (the Nordic specialist most likely to handle this reliably) reportedly charges a one-time alphanumeric sender ID registration fee around €380+VAT covering all Norwegian operators — a real cost before the feature even works once.
- **Verdict: defer.** Zero free allowance, linear per-message cost, plus a non-trivial one-time setup fee to make links actually deliver — none of which is justified while there's no paid tier (app is still "track referrals now, reward later").

**Abuse/spam risk (applies even to the cheap email version):** letting a user type someone else's email or phone number and triggering an automated send to them is a well-known "send-to-a-friend" abuse vector (also usable for harassment or enumeration attacks). Needs, before shipping even the email-only version: rate limiting per sender/IP, CAPTCHA or honeypot as a backstop, sender must be logged-in/verified (ties abuse to an identifiable account), a fixed/templated message body (never user-controlled free text, to protect the sending domain's reputation), recipient opt-out/no-further-contact handling, and an audit trail (who sent what to whom, when — useful for abuse investigation and any GDPR/Norwegian Marketing Control Act questions, since the recipient's personal data is processed without their own consent at point of entry).

**How to implement (if/when prioritized):** start with Resend + Flask-Mail (SMTP) for email-only sending, add the abuse guardrails above before launch, and treat SMS as a separate, later decision gated on having a paid tier to justify the cost.

**Decision (2026-06-29): NOT PURSUING — closed.** User asked specifically about two DIY alternatives: (1) sending SMS from their own personal phone number "hidden behind" a business name like "Menu Planner," and (2) self-hosting a mail server on their own Raspberry Pi/home internet connection, since they have free messages/data in their phone subscription. Both ruled out:
- **Personal phone "hidden behind a name" doesn't work the way it sounds** — masking a sender as a business name (an "alphanumeric sender ID") is a carrier-level feature tied to a registered business SMS account, not something applicable to a personal SIM. A personal number is always just that number to the recipient. Phones also aren't built for programmatic bulk sending at any real volume — that path leads to account bans, not a real feature, and still costs per-text past whatever's "free" in the plan.
- **Self-hosted mail server on a home connection is a well-known bad idea**, not a viable cost-saver: residential IP addresses are blacklisted by major mail providers by default (the single most common spam pattern), so emails would mostly land in spam or get silently rejected by Gmail/Outlook/etc. Fighting that deliverability problem would cost far more time than Resend's free tier already saves at $0/month.
- **User's own fallback idea — just let people copy/paste the link themselves — is the one being kept.** This is already fully built (the referral link in Settings, the `/welcome` landing page with demo videos) and sidesteps every problem above: no API cost, no carrier filtering, no "type someone else's contact info into a form" abuse vector, since the person sharing it is the one choosing to send it via their own text/email/social app. Revisit automated sending only if there's ever a concrete reason the manual-copy approach isn't enough.

### 17. Import a recipe from a URL, rewritten into our own ingredient/steps format (research only, not started)
**Background:** user asked whether the app could let someone paste a recipe URL (test case: matprat.no, a major Norwegian recipe site), fetch that page, and rewrite the ingredient list and steps in different words to pre-fill the "Add New Recipe" form — reasoning that bare ingredients/instructions aren't copyrightable, so rewriting them should be legally fine. A research agent investigated this specifically.

**The user's copyright theory is correct, but incomplete — there's a separate legal problem it doesn't cover:**
- **Copyright law (US, EU, and by extension Norway) genuinely does NOT protect bare ingredient lists or plain step-by-step instructions** — these are treated as "facts"/an uncopyrightable "process." What IS protectable is the specific creative *expression* layered on top (narrative flourishes, personal commentary, photos). Genuinely rewriting instructional prose (not a thin synonym-swap) generally does avoid copyright infringement on the expression, and rewriting the ingredient list isn't even necessary since a bare list was never protected to begin with. **So the user's theory is right as a matter of pure copyright doctrine.**
- **But matprat.no's Terms of Use is a separate, contractual restriction that doesn't care what copyright law says.** Their terms (`matprat.no/retningslinjer-for-bruk-av-innhold/`), translated directly: *"it is not permitted to borrow, copy, or extract videos, recipes, or other content directly from matprat.no without written permission"* and *"it is not permitted to extract large portions or the entirety of our recipe database, for use in apps and websites."* That second line is close to a verbatim description of this feature. **Rewriting the words doesn't fix a Terms-of-Service violation** — ToS is a contract claim, independent of whether the underlying content is copyrightable (this is settled doctrine, e.g. *Ryanair v. PR Aviation*, CJEU 2015: facts/data can be contractually off-limits via ToS even when copyright law wouldn't protect them).
- **Third, smaller risk layer:** the EU/Norway "database right" (implemented in Norway's Copyright Act §24) protects a site's *collection* of recipes as a database, separate from whether any single recipe is copyrightable. Low risk for one-off single-URL use, but scales up as a real risk if many users repeatedly pull from the same site through this feature.
- **matprat.no does not publish Schema.org/JSON-LD Recipe structured data** on its pages (checked directly), so there's no clean "the site opted into reuse" signal for this specific site either — extraction would require brittle HTML scraping, which is also a weaker legal position than reading structured data a site intentionally publishes for reuse.

**Verdict:** **Don't build this against matprat.no without permission.** Their terms explicitly prohibit exactly this use case, and that's a binding restriction regardless of the rewriting/copyright angle. Two real paths forward if this feature is still wanted:
1. **Ask matprat.no directly** — their terms literally invite emailing `post@matprat.no` for written permission. Free to ask, no idea if they'd say yes.
2. **Build it generically against sites that opt in via Schema.org/JSON-LD Recipe markup** (many international food blogs publish this specifically so tools can read it) — fundamentally different and safer legal posture than scraping a site whose terms forbid it. This wouldn't cover matprat.no itself, but would work for a broad set of other recipe sites without needing per-site permission.

**How to implement (if pursued via path 2):** fetch the URL server-side, look for an `application/ld+json` block with `@type: Recipe` (the standard schema.org format), parse `recipeIngredient` and `recipeInstructions` from it, then optionally pass through an LLM to rewrite/reformat into the app's own "Add Recipe" form fields. Skip sites without this structured data rather than falling back to HTML scraping, since that's where the legal and reliability risk both increase together.

**Second research pass (2026-06-28) — alternatives that don't require waiting on matprat.no's permission:**

- **Photo-to-recipe (upload a photo, app extracts ingredients/steps) — recommended, build this first.** A single Claude vision API call with the photo + a request for structured JSON output (ingredients array, steps array) handles messy/handwritten/cookbook photos far better than classic OCR (Tesseract scored near-zero on table-like layouts in benchmarks; a vision LLM does extraction and structuring in one step). Cost is trivial — well under $0.01 per image at Sonnet pricing, even cheaper at Haiku; 1,000 users adding 5 recipes each via photo is roughly $15-30/month at Sonnet, under $10/month at Haiku. Effort is a day or two: one new Flask route, one Anthropic SDK call, wire the JSON result into the existing Add Recipe form as a pre-fill the user reviews before saving.
  - **Legally meaningfully safer than URL scraping**: there's no website Terms of Use being violated at all here — the user is digitizing a recipe they already physically possess (their own cookbook, a handwritten family card) for their own private household collection, not republishing someone else's website content. Same legal territory as a user manually typing in a recipe from a cookbook they own, which the app already allows — photo-to-text doesn't introduce new risk beyond what manual entry already exposes the product to.
- **Pre-made recipe packs from third-party APIs — mixed results, most are dead ends:**
  - **Spoonacular and Edamam are both disqualified** — their terms explicitly and unambiguously prohibit permanently storing their data in your own database (only narrow, short-term caching is allowed, sometimes only with separate written permission). Using either for "baked-in packs" would recreate the same category of legal risk as the matprat.no situation, just with a different counterparty. Only viable as a live, per-request paid lookup feature (different feature, ongoing subscription cost, not worth it for a free product right now).
  - **TheMealDB explicitly permits storing/copying their API data** (with attribution, API-only access, and a modest fee required before publishing an app that uses it) — the one real option here, but Nordic/Norwegian dish coverage is unconfirmed and would need manual spot-checking before relying on it; likely thin since it's a general international database. Could still be useful to pad out the existing "European Classics" pack.
  - **CC BY-licensed datasets (e.g. the OpenRecipes project)** are legally clean for storage but US-centric and dated (2017), low practical value for a Norwegian audience without heavy curation/translation.
  - **Norway-specific sources**: only found Matvaretabellen (Norwegian Food Safety Authority, CC BY 4.0, has a public API) — but it's a nutrition/food-composition database, not a recipe collection; its "recipes" are nutrition-calculation artifacts for composite dishes, not real step-by-step cooking instructions. Would need substantial rework to become consumer-facing, and is a separate idea (nutrition-facts-per-ingredient) more than a recipe-pack source.
- **Updated verdict:** ship photo-to-recipe as the near-term unlock for "I don't want to type this in" — it doesn't need to wait on matprat.no's permission at all. Keep pursuing matprat.no permission separately for authentic, large-volume Norwegian content longer-term. Treat "more packs via TheMealDB" as a narrow, low-priority follow-up gated on confirming it actually has useful Nordic content first.

### 30. Edit Recipe save always showed "Error: undefined", even on success — found and fixed 2026-06-29
- **Reported by user:** editing "Spaghetti Bolognese" (changing "heat" → testing the new one-line ingredient format) and saving produced "Error: undefined" — but the change later appeared to have actually been applied.
- **Root cause: a total mismatch between frontend and backend response shapes**, confirmed by reading both sides directly. `/api/edit-recipe` (`pi-deployment/flask_app.py`) returns `{'status': 'success', 'message': '...'}` on success or `{'status': 'error', 'message': '...'}` on failure — it has never had a `success` boolean field or an `error` field, only `status` and `message`. But `edit_recipe.html`'s save handler checked `if (result.success)` and, on the else branch, read `result.error` — both fields that have never existed on this endpoint's response. This means the JS **always** fell into the error branch regardless of what actually happened server-side, and always rendered the literal text "Error: undefined" (`result.error` being `undefined`) — explaining exactly why the user saw an error on a save that had, in fact, succeeded. This bug has almost certainly existed since the edit-recipe feature was first built, silently making every successful save look like a failure.
- **Fixed:** changed the check to `if (result.status === 'success')`, and the error path to read `result.message` (falling back to "Unknown error" if that's somehow also missing) instead of the nonexistent `result.error`.
- **Verified end-to-end:** a real successful save now correctly reports success; a real failure (tested with a nonexistent recipe ID) now shows the actual, useful server message ("Recipe does_not_exist not found") instead of "undefined".

### 34. Where does feedback actually land? — led to finding a real Railway data-persistence gap, fixed 2026-06-29
- **User asked, after submitting a real test feedback entry:** where does the submission actually go?
- **Answer:** lands in `data/feedback.json`, a plain JSON file on disk, exactly as designed in item 28. Locally that's a real file on the user's own machine; on Railway it would be the same relative path inside the deployed container.
- **Found a real, more serious problem while answering this:** Railway's container filesystem is ephemeral by default — every redeploy (which happens on every future `git push`) rebuilds the container from scratch, wiping anything written to its local disk. This isn't just a feedback-system problem: `data/feedback.json` AND every per-household JSON file (`data/households/<id>/pantry.json`, `recipes_db.json`, `categories.json`, `activity_log.json`, etc.) would silently disappear on the next deploy once this is live on Railway for real testers — confirmed no persistent volume is configured anywhere in `Dockerfile`/`railway.json`.
- **Fixed:** guided the user through adding a Railway persistent volume (done via Railway's own dashboard — Settings → Volumes → mount path `/app/data` — not something doable from code/git, since it's an account-level infrastructure setting). Separately, found and fixed a real follow-on risk: a freshly-attached Railway volume mounts **empty**, which would shadow/hide the seed data files (`categories.json`, `pantry_staples.json`, `sample_recipes.json`, the 5 recipe packs) that are normally baked into the Docker image at `/app/data` — meaning the first boot against a new volume would have no seed data at all. Fixed in `Dockerfile` (keeps a pristine copy of the baked-in seed files at `/app/data-seed`, separate from the path the volume will shadow) and `docker-entrypoint.sh` (restores any seed file genuinely missing from `/app/data` on container start, via `cp -rn` — no-clobber, so it only fills gaps and never overwrites real data from a previous boot). Verified the no-clobber behavior directly: simulated a fresh volume with existing real user data (`feedback.json`) and missing seed data (`categories.json`) — confirmed the missing file gets restored while the real data is left completely untouched. Confirmed this change is a complete no-op for local development (the `/app/data-seed` directory only exists inside the Docker image, never locally), so nothing changes about how local testing works.
- **Still needed (manual, on Railway's dashboard, not done by this fix):** the user needs to actually add the persistent volume in Railway's UI before this protection takes effect — the Dockerfile/entrypoint changes are necessary but not sufficient on their own; without the volume itself being attached, the filesystem is still ephemeral regardless of the seed-restore logic.
